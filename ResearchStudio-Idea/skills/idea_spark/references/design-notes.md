# Idea Spark — Design notes (rationale, not runbook)

This file records WHY the pipeline is shaped the way it is. Nothing here is needed to *run* the skill — SKILL.md and the per-phase system prompts carry every operational contract. Read this when modifying the skill, evaluating a design change, or debugging a quality (not mechanics) problem.

## Design principles

1. **Innovation Patterns are diagnostic vocabulary judged per-gap, not classification labels OR generative templates.** The 15 induced ideation patterns (reframe-as-solvable-object, assumption-audit-and-pivot, algebraic-equivalence-unification, heterogeneous-decomposition, architectural-operator-substitution, structural-prior-encoding, characterize-limit-then-surpass, self-supervised-signal-engineering, targeted-self-supervised-objective, controlled-diagnostic-design, unify-into-shared-representation, adapt-via-conditioning, generative-process-redesign, decompose-and-delegate, relax-discrete-search-to-continuous) are how the corpus describes productive research moves. Phase 2.1 reads each pattern's **Definition + Operational signature + When to apply** panels and judges per-pattern per-gap whether the pattern's move closes the gap. Phase 2.2 picks ONE sub-pattern under each chosen main pattern by reading its `tactical_pattern` + `Step-by-Step` + `when_to_pick_this_one` + `differentiation_within_parent` panels at generation. The sub-pattern's Step-by-Step is 5 abstract steps distilled from the cluster's [Accept] examples — domain-agnostic structural moves with [Reject]-derived boundaries embedded, written WITHOUT paper-ID citations so the candidate-author applies the abstract pattern to their own gap rather than mimicking specific papers. Treating patterns as generative templates (verbatim recipe execution + siblings_considered + lock-in rules) converged generation toward corpus-validated incremental work; the cognitive-tool framing avoids that.

2. **The anti-incremental commitment is ≥2 ideation patterns per CANDIDATE, composed the way the corpus composes — not ≥2 gaps.** The corpus's own per-paper tagging (1-3 patterns each) shows multi-pattern Orals compose moves in SERIES on ONE contribution: pattern A's move produces the object pattern B's move operates on. Phase 2.1 therefore realizes the ≥2-pattern default preferentially as a CHAIN on the anchor gap (second pattern from the attested `companion-combos.md` set, with a named intermediate object), and only exceptionally as a PARALLEL composition via a sibling gap — which must EARN its seat through the removal test ("the anchor's story is incomplete without it": falsification weakens, an obvious reviewer objection opens, or the claim loses verifiability). Anchor-only selection is legitimate and common (1-3 gaps total); a single-pattern selection is permitted only with an explicit `composition_note` defense (deterministically presence-checked, audit-weighed). History: an earlier design QUOTA-forced 2-4 gaps with randomly-sampled siblings admitted by a coherence-thread test alone; that test was structurally vacuous (all gaps descend from one bottleneck, so `shared_question` was satisfiable by construction), the prompt escalated to resampling batches and even re-picking the anchor when no sibling passed, and in validation runs the audit's reject-lesson borderlines concentrated exactly on the forced sibling entries — the Frankenstein signature domain experts read as "makes no sense". Saturation-aware preference is unchanged (avoid saturated and untested lanes; prefer mid-frequency; saturated picks need a novel-angle defense at audit). Phase 2.2 enforces substantive novelty via `differentiation_from_lit[].delta` (what we derive/claim/construct/measure that closest_adjacent did not — not "we use a different ideation pattern").

3. **Contribution shape is signature-agnostic; there is NO mandated theory+engineering dual-leg.** (An earlier draft mandated both legs, but no executing surface ever enforced it — the candidate schema has no theoretical_leg field, no audit check reads one, and Phase 4's `feasibility_validation.theoretical` explicitly allows `n/a`. Mandating it is the same forced-composition pressure that produced stapled siblings, so the mandate is retired rather than enforced.) The theoretical component is driven by `intake.contribution_type`; a contribution can be theorem OR observable regime OR scaling exponent OR measurement primitive OR architectural property — the audit doesn't prefer one Oral signature (theory + reframing-first) over others (scaling-law, empirical-reveal, surgical-fix, benchmark-validity). The anti-sketch guarantee that the dual-leg mandate was meant to provide is carried by three enforced gates instead: `falsification_structure_check` (audit check 5), the Phase 2.3 numeric dry-run, and the 4.1.5 implementability audit. All Oral shapes the corpus contains can score 5.

4. **Mechanism-aware falsification.** Every candidate's `falsification_prediction` is a single paragraph (3-5 sentences) that visibly contains (a) the minimal experiment, (b) which metric moves and in which direction if the candidate works (name the metric + qualitative direction; the experiment establishes the magnitude), and (c) a mechanism distinguisher pivoting on ONE NAMED LOAD-BEARING VARIABLE — the single quantity (e.g., a gradient norm, an information-gain term, a logit divergence, a learned threshold, a representational direction) whose behavior carries the mechanism claim — plus a negative-control intervention on that variable that should drive the DOWNSTREAM OUTCOME METRIC back to baseline. The negative control's predicted effect MUST be the task-outcome metric that defines the mechanism's value (accuracy, regret slope, refusal pass-rate trajectory) — NOT the load-bearing variable's own value or any quantity analytically derived from it (a control of the form "intervene on X → X becomes 0" tests a definition, not a mechanism). A positive control (a stripped-down model using only the load-bearing variable that recovers most of the downstream effect) is recommended when feasible. Without the load-bearing-variable-plus-non-tautological-negative-control structure, "metric moved" remains consistent with calibration improvements / estimator quality / data shifts / many other non-mechanism explanations — and the candidate is the dominant Reject signal in the corpus. `compute_budget` is a separate flat field, **user-relative** (no absolute cap), carrying BOTH cost lines when applicable (GPU-days + inference/API dollars for the full falsification campaign) — Phase 4 feasibility_validation buckets each line against `intake.compute` and takes the worse. **Factory default `intake.compute = 80GB-class GPUs (A100/H100), up to 8 concurrent (one node), ≈150 GPU-days over 5 months, plus ~$10k inference/API campaign budget`** — the canonical "funded single researcher" scale. Why this shape: the unit is a VRAM class rather than a chip model so it doesn't stale-date; the concurrency line exists because "1 GPU × 5 months" and "8 GPUs × 19 days" have equal totals but different feasibility (aggregate-memory-bound training needs the node); the API line exists because LLM-era falsification campaigns (closed-model baselines, LLM judges, long reasoning traces) are often bounded by the API bill, not GPU-days — an implicit $0 default made feasibility judgments on those ideas meaningless. The magnitude is deliberately NOT lab-scale: the default is the yardstick for users who stated nothing (median: funded student), and well-resourced users are exactly the ones able to state their budget. Precedence: user query > `IDEASPARK_DEFAULT_COMPUTE` env (standing per-user profile, auto-loaded from `.env`, surfaced by `next` to Phase 1) > factory default. Both `falsification_prediction` and `compute_budget` are kill-switch fields: byte-identical preserved across Phase 2.2 → Phase 3.3 (when revise runs) → Phase 4 — with exactly one audited exception, see "The falsification rewrite exception" below.

5. **Anti-pattern is empirical negative knowledge — audit-only.** The corpus identifies 3 reject-favored 2-way compositions (audit + auxiliary_signal, audit + invariance, audit + surgical_fix), each with a specific required_mitigation; rates and mitigations live in `references/anti-patterns.md`. **Phase 2 does NOT load anti-patterns.md** — naming reject-prone compositions during generation creates Streisand-effect priors that bias selection. Phase 3.2 audit's `anti_pattern_check` reads anti-patterns.md, detects matching compositions via the SET of `gap_closure[].main_pattern` values, and verifies substantive mitigation delivery (not keyword presence). Failed audit → Phase 3.3 revise rewrites the candidate's affected fields with the corpus-grounded fix.

6. **Cheap kills first, expensive expansion last.** Phase 3 runs collision retrieval (real, ~30s, no LLM) before audit (single LLM call replacing earlier 4-attacker simulation). Heavy expansion of the candidate into motivation + method_flow + claims + abstract happens only in Phase 4, after the candidate clears the gauntlet.

7. **Phase 3.2 is judgment, not modification.** The audit reports what corpus evidence triggered which signals; it does NOT auto-revise the candidate. When revision is needed, the audit emits `revision_targets[]` and Phase 3.3 (a separate LLM call) applies them — keeping audit and modification on different surfaces avoids the self-answering bias of cherry-picking attacks one can already answer.

## The falsification rewrite exception (why the kill switch has one door)

The kill-switch contract exists to stop SILENT drift: a reviser or expander quietly substituting an easier experiment or a smaller budget. But the audit's `falsification_structure_check` can find the paragraph *structurally* deficient (no named load-bearing variable; a tautological negative control that predicts the variable's own value instead of the downstream outcome metric) — and that deficiency is exactly the dominant Reject signal the field was designed to avoid. A rule of "auditable but unrepairable" would force `abandon` on candidates whose experiment and claim are fine and only the falsification *writing* is broken.

So the contract is "no unaudited change", not "no change": Phase 3.2 (and only it) may emit a `scope=falsification` revision_target; Phase 3.3 applies it via the dedicated `rewrite_falsification` op; the merger verifies the audit's authorization (`--critique`) and stamps `falsification_rewritten`; a bounded single-check re-audit must then pass before Phase 4; and `kill_switch_integrity` re-bases the byte-identity anchor at the Phase 3.3 final_candidate. The reviser cannot self-authorize, the rewrite cannot change experiment/metric/claim, there is at most one attempt per run, and `compute_budget` has no door at all.

## The bounded internal retry (why abandon isn't immediately terminal)

Phase 2.1's sibling-gap sampling is random. A single `abandon` verdict therefore often reflects an unlucky draw (a gap×pattern combination that walks into a documented Reject lesson), not an unworkable direction. The one-shot guarantee constrains *asking the user*, not internal regeneration — so the host archives the failed attempt under `attempt_1/`, marks `.retry_used`, and re-runs Phase 2.1+2.2 exactly once with the failed attempt's audit + selection as negative constraints (see the OPTIONAL retry input in `ideate_select.txt`). A second abandon writes `phase_3_failed.md` citing both attempts. Phase 0/1 artifacts are reused; the retry costs only the generation + gauntlet calls it re-runs.

## Why Phase 2.1 + 2.2 run in ONE sub-agent (but 3.2/3.3 and 4.fill/4.1.5 never merge)

The call-separation rule protects *adversarial* surfaces: the LLM that proposes attacks must not be the LLM that answers them (3.2 vs 3.3), and the persona that authored the method must not be the one auditing its implementability (4.fill vs 4.1.5). Phase 2.1 → 2.2 has no such boundary — both are generation-side, and 2.2 consumes 2.1's output directly. Running them in one sub-agent context (writing BOTH output files, so every audit anchor is preserved) saves a sub-agent spin-up plus a duplicate read of phase1 output + intake, with no bias cost. The two-STAGE design (select at Operational-signature level, then generate with sub-pattern cards open) is unchanged — it is about judgment layering, not about process isolation.

## Why the coherence gate executes instead of reviewing (and why it is narrow)

No pre-2.3 quality surface checked the candidate's INTERNAL procedural validity: the citation gate checks provenance, the 3.2 audit checks external facts (corpus Reject patterns, prior art, falsification structure), and 4.1.5 specifies-but-never-modifies at Phase 4 — too late and too constrained. A real run supplied the motivating bug: the 2.2 candidate's "resample R=16 times until the 24-bit parity matches" reads fluently and is vacuous (acceptance probability ~4⁻²⁴); the audit caught it only incidentally. Logic bugs read fluently by construction — they surface only under EXECUTION — so the gate's core action is a numeric dry-run on a small concrete instance (plus dataflow formalization, degenerate probes, claim→step mapping), not a checklist read.

Two failure modes shape the contract. (1) Self-answering bias: the context that wrote the bug rubber-stamps it, so 2.3 is always a FRESH context (same reason 3.2/3.3 and 4.fill/4.1.5 are separate calls). (2) Refine-to-blandness: open-ended "improve this idea" passes sand the edges off (the same convergence-to-incremental force documented under "Why 2 stages with judgment"), so the gate is single-pass, closed-scope (repairs only what a specific T1-T5 finding requires), and forbidden from the novelty surface, pattern bindings, and kill-switch fields. Unfixable-without-redesign findings are recorded, not "fixed" — the audit's abandon/retry machinery owns redesign decisions. The audit stays blind to the 2.3 report so "already checked" cannot soften it. Verification ladder this slots into: paper-level soundness (2.3, necessary not sufficient) → provable claims (theory leg, established by derivation) → empirical claims (only the falsification experiment).

## Why the naive-baseline audit exists (the depth mechanism)

A paired comparison exposed the hole. Two candidates were generated for the same bottleneck (DSpark serving-time statistics) from the same phase0/phase1: the quota-era pipeline produced a candidate whose mechanism confronted scheduler-induced CENSORING (the scheduler chooses what gets observed FROM the very estimates being calibrated → naive averaging is selection-biased → propensity weighting + ε-exploration) — an external review rated it top-venue material precisely for that machinery; the refactored pipeline produced a cleaner single-story candidate whose mechanism was EMA + shrinkage, censoring never confronted — "why not just EMA?" has no answer, and the review rated it incremental. Post-mortem, three facts located the flaw: (1) phase1 carried ZERO censoring content — the insight was invented at generation time under the assumption_audit pattern that the QUOTA had accidentally forced into the composition, so the old depth was a side effect of the very defect the refactor removed; (2) every quality criterion in the refactored selection/gate stack is a SATISFICING criterion ("is the story complete", "does the procedure survive") — the cheapest complete story always wins, and EMA is complete; (3) the pipeline's own coherence trace TOUCHED the censoring shadow (cold buckets with 0/0 estimates) and patched AROUND it with abstention, because the repair contract said repair-don't-redesign and nothing distinguished a patch-hole from THE contribution-hole.

The fix is a demand-verify pair at the mechanism level (NOT at gap selection — the reviewer's own verdict placed all the value inside one mechanism story, so re-legitimizing sibling stapling would fix nothing): Phase 2.2 must answer the reviewer's universal existence question — "what is the naive version, and why doesn't it already work?" — with three honest branches: (i) the naive version silently relies on a premise false in this setting (usually an OBSERVATION-MODEL premise: how inputs are sampled/observed — censoring, selection bias, non-iid, leakage, self-preference all live there; a mandatory observation-model premise was added to the ledger for this reason) → confronting that failure IS the contribution; (ii) naive suffices and the field knows → an incremental signal surfaced honestly instead of decorated over; (iii) naive works but the field disbelieves it → counter-intuitive minimalism, requiring evidence of the belief (hook shape (c)'s admission test). Branch (iii) exists because some Orals ARE simple-and-right; a two-branch version would systematically kill them. Phase 2.3 then verifies by execution (T5): it constructs the naive version INDEPENDENTLY (the candidate's own stated naive can be a strawman), runs it on the same dry-run instance, and classifies `confronts_obstacle` / `equivalent_to_naive` / honest `n_a` (pure-equivalence shapes degrade to tracing where the standard argument breaks). `equivalent_to_naive` and any finding coinciding with the declared obstacle are OBSTACLE HOLES: avoidance-style patches are forbidden for them, they surface as blocking, the audit treats them as load-bearing (revise to confront, or abandon), and the internal retry carries the obstacle as a POSITIVE directive. Honest limits: this is a depth DETECTOR and redirector, not a depth generator — directions where naive genuinely suffices are correctly reported as incremental rather than inflated; and 2.2's self-answer can still be gamed, which is why T5's independent construction is the enforcement layer (the same demand-then-verify-by-execution pattern as everything else in this skill).

## The universality criterion (what changes are allowed to ship)

Every constraint added to the generation side is paid by EVERY future idea, so the admission bar for skill changes is: net effect ≥ 0 for every idea shape (method, generative/perception, systems, theory), with "true zero cost when n/a" as an acceptable form of ≥ 0 — a conditional clause that fires only when its precondition holds costs nothing where it doesn't apply. Changes that help one shape and hurt another do not ship until reformulated conditionally (this batch's examples: the observation-model premise and the stakes clause were both born as unconditional demands that would have forced theory-shaped ideas to fabricate practitioner scenarios and sampling stories; both were made conditional with explicit honest n/a escapes — fabrication to satisfy a checklist is strictly worse than the missing field).

Two constraint families behave differently under this criterion. GENERATIVE constraints (forced pattern counts, forced sibling gaps, forced mechanism families) throttle the generator — the quota era proved this empirically — so the generation side takes only minimal honest-declaration demands (a ledger, an audit paragraph, claim wording at defensible strength: net a few sentences). HONESTY constraints (grade this claim, compare against the naive version, execute the dry run) do not throttle, because a truthful candidate passes them for free; their teeth therefore live on the VERIFICATION side (2.3), which grades and repairs wording but is forbidden from redesigning. The complement rule keeps honesty constraints from backfiring into blandness: grading must never reward claim-avoidance — a strong claim with honest stated assumptions outranks a hedged weak claim, and an all-hedged candidate is itself flagged as asserting nothing checkable.

Claim grading is SEMANTIC, not lexical: T4 classifies each asserted property as established / conditional / overclaim / empirical by arbitration matched to claim type — statistical claims settled by an executed Monte Carlo (the measured number decides, e.g. an SNIPS-style "finite-sample unbiased" claim dies to a measured bias, not to keyword matching), theorem-shaped claims recorded as proof obligations (at best conditional until proved), everything else by the T1 mapping argument. Execution beats estimation for the same reason 2.3 executes at all — a fresh-context hand-walk can still hallucinate arithmetic — so T2/T5 write and RUN a stdlib script when an execution tool exists, degrading honestly to `unexecuted` when not.

The retry budget follows the same criterion, stated as ONE rule instead of a death-type taxonomy: a retry is justified only by INFORMATION GAIN — the failed attempt produced binding lessons (upheld executed findings, new subsuming papers, new triggered corpus lessons) that the previous generation did not have. The empirical basis is uniform across recorded runs: directive-carrying retries converged in one attempt (the inert-IPW→ε-exploration and vacuous-IPCW→PACE repairs); blind re-rolls under unchanged constraints did not. The two former special-case branches fall out as instances (first abandon always retries because generation ran with zero audit information; a subsumption lesson REPEATED across attempts binds at the framing level, so the answer is re-diagnosis, not another roll), and combinations nobody wrote branches for are handled by the same comparison — e.g. an obstacle death followed by a fresh subsumption threat retries on the new negative anchor, while a second death repeating an already-seen lesson terminates immediately. A global cap (3 candidate cycles per framing + 1 re-diagnosis with 1 granted attempt) bounds cost at 4 gauntlet cycles; the cap is the only non-informational element, and it exists because even informed retries have a price.

Regression protocol: because the generator is stochastic, "did the same idea come back" is meaningless; regressions are judged at the PROPERTY level over a fixed cross-shape direction set (references/regression-directions.md, deterministic subset in scripts/regression_check.py). The theory direction is mandatory in every sweep specifically to exercise the n/a escapes without fabrication — it is the shape most easily hurt by well-meant universal demands.

## Why 2 stages with judgment, not lock-in

A lock-in alternative would demand verbatim quotes from cards + enforced siblings_considered + sub-pattern recipe execution + many hard rules; cumulatively that converges generation toward corpus-validated incremental work and kitchen-sink mechanism stacks. The judgment-based 2-stage design (Phase 2.1 = judgment per pattern at Operational-signature level; Phase 2.2 = sub-pattern as descriptive vocabulary not recipe) produces paper-shape candidates while preserving audit anchors via `gap_closure[].main_pattern + sub_pattern`.

## K=1, not K=2/3

Single candidate goes through critique. (Earlier K=3/K=2 design had no auto-selection downstream — overhead without quality win.)

## Audit check history

Earlier design had a check `almost_prior_factcheck`. Removed: low fire rate, redundant with paper_pointed_threat. A `saturation_defense_check` was also tried and removed: it was the one purely advisory check — soft signal only, never a hard floor, and unlike the others it consulted no retrieved related work — so its concern now folds into Phase 4's `reviewer_concerns_and_responses` instead of gating Phase 3.2. Saturation metadata still flows: Phase 1 computes the band, Phase 2.1 records it, Phase 4 echoes it into `domain_landscape`. `recipe_application_check` was added because the deterministic citation guard can only prove parent-consistency, so a recipe built from the parent's gist while citing a real cluster passes the guard yet bypasses the cluster's actual tactic; this check is the semantic backstop. `falsification_structure_check` is the newest (fifth) check — before it, the falsification paragraph's structure (design principle 4) was demanded at generation but never audited, and the kill-switch made it unrepairable; now it is audited, and repairable through the single-door rewrite exception above.

## Why the verdict is two-layer

Pure mechanical aggregation over-triggers revise (treats 1 trivial borderline same as 3 severe). Pure LLM verdict introduces agreeable-bias and loses audit trail. Hard floor preserves non-negotiable corpus facts; soft layer uses context to distinguish "must fix" from "noted concern, advance".

## Why Phase 3.3 is patch-only

Previous versions of the 3.3 contract had the LLM echo the full candidate back with edits applied; a single ~25k-token candidate echo caused a real backend inference timeout (the kill-switch fields, the largest, were re-typed verbatim). The patch + deterministic merger design removes that class of failure and makes the anti-substitution contract structural: the model physically cannot write a guarded field (outside the audited falsification door).

## Why Phase 4 is skeleton → fill → assemble

Phase 4's full expansion JSON has ~30 top-level fields and ~half are mechanical transforms — kill-switch echoes, venue_year lookups, group-bys over `lit_table.md`, joins of `gap_closure × pattern_saturation`, reviewer-concern lifts from the audit report. Asking the LLM to re-type those wastes tokens and risks a backend inference timeout (the same shape that broke Phase 3.3 before the patch-only redesign). The deterministic skeleton populates every mechanical field; the LLM authors only the ~12 prose TODOs (~8k instead of ~20k tokens); the deterministic assembler merges and refuses kill-switch writes.

## Why Step 4.1.5 exists (and is compute-agnostic)

Phase 4.1's `method_flow.steps[]` are often too terse to *understand* — a step names an operation ("extract premises", "score consensus", "train a critic") without the concrete object it runs on, so the method "reads" but cannot be built from. 4.1.5 adopts a fresh, skeptical implementing-engineer persona (separate call from the 4.1 author — same anti-self-answering rationale as principle 7) and rewrites each step into a specification an engineer could code from. It MUST NOT consider compute/wall-clock — resource feasibility is judged separately by 4.1's `feasibility_validation`; conflating the two re-introduces exactly the hand-waving the step removes. A step that is expensive but fully specified is the correct output.

## Why collision is dual-channel (signature@10mo + alias@48mo)

Phase 0 covers 0-24mo but with broad-domain TOPIC queries; Phase 3.1 re-queries by the candidate's mechanism-specific `signature_terms[]`. With the original 6-month collision window, a same-mechanism paper 7-24 months old could be missed by both passes (too old for collision, too mechanism-specific for Phase 0's topical recall). 10 months narrows that at negligible cost.

But a real run exposed a second, LEXICAL blind spot no window widening can fix: the same mechanism under another community's name. A "goal-image conditioned scorer for task completion" (VLA vocabulary) had 2023-2024 near-ancestors published as "goal-conditioned success detectors" / "goal-image reward models" (reward-modeling vocabulary) — signature terms drawn from the candidate's own wording never contain those tokens, so a 48-month signature search still returns nothing. The fix uses the one knowledge source retrieval lacks: Phase 2.2 emits `alias_terms[]` from PARAMETRIC knowledge ("what would other communities call this?"), and Phase 3.1 runs them as a second channel over a 48-month window. Hits carry a `collision_channel` tag; the audit is told not to discount alias-channel threats for age (an ancestor subsumes regardless).

Dual-channel retrieval roughly 2.4×'d the raw pool (a real run hit 653 hits / 721KB — beyond what the audit context can read whole), so the orchestrator relevance-truncates the audit-facing file: each hit is scored by lexical overlap between its title+abstract and its OWN channel's query terms (crude prefix-stemmed content-word match — deterministic, zero deps); zero-score hits are dropped unconditionally (they are BM25 phrase-fragment accidents — a true same-mechanism paper is retrieved BY those terms, so its overlap cannot be 0); each channel then keeps its top-120. Abstracts are slimmed to 1000 chars. Drops are printed per channel with the min kept score (no silent caps), and the untruncated pool is preserved as `collision_hits.full.json`. Embedding/LLM reranking was considered and rejected for this job: reranking a fixed pool where the true positives share vocabulary with the query terms by construction is exactly the case lexical overlap handles; heavier rerankers buy nothing here (they matter for RECALL improvements — e.g. a future citation-graph third channel — not for this cut). Backstop for families neither channel surfaced: the audit's `parametric_family_concern` soft signal names the family + query vocabulary (never specific papers — family names don't hallucinate, cites do) and flows into Phase 4's reviewer_concerns as a "run a scoop-check on X before investing" flag rather than gating the verdict.

## Why Phase 0 retrieval is orchestrated (not WebSearch)

Skills are advisory — when SKILL.md says "run Phase 0 literature search", the model has multiple paths (Skill tool, direct Bash, WebSearch simulation, fetch arxiv URL), each "satisfying the spirit" of Phase 0. Soft rules don't reliably prevent tool drift. The orchestrator collapses the choice space: SKILL.md Phase 0 says "run THIS Bash command" — the model has one path or must explicitly admit failure. Coupled with Phase 1's entry assertion (lit_grounding_mode + retrieved_via), bypass becomes mechanically detectable, not just discouraged.

## Phase 0 pool design

Role-based retrieval: each connector used where it's most informative (arxiv 0-6mo preprints / openalex + semanticscholar 6-24mo published / openreview 0-6mo in-review), non-overlapping windows so cross-source dedup on (title_norm, externalIds) doesn't double-count, ~40-45 paper target, SS-first dedup priority (its externalIds block is the best cross-source anchor). Full-text fetch caps the pool at the most relevant ~15 (+ user refs, never capped), method-first ordering (eval/benchmark-only papers sink), concurrent fetch with per-paper budget so a slow PDF can't stall the step, HTML path first (~85% of 2024+ ML preprints) with pymupdf PDF fallback. Limitations sections are intentionally NOT extracted — author-written limitation paragraphs are often weaker than what the audit synthesizes from method + experiments.

## Output surface policy

No calendar projections (sequencing in dependencies, not weeks). No experiment matrix / ablation plan / baseline table / expected figures — the skill produces IDEA + falsifiability + feasibility judgment; experimental engineering is the user's responsibility. Echo vs reference: anti-substitution-guarded fields and structural lookups are filled by the deterministic skeleton; `closest_adjacent` and `lit_grounding_mode` are rendered directly by the card template, not duplicated into Phase 4 output.

## Why blocking findings reach the audit as a file, not narrative (and why the guard checks presence, not quality)

Origin: a live run where the coherence gate EXECUTED a refutation of the candidate's core machinery (equivalent_to_naive, measured numbers) and the audit — reasoning only, never shown the findings — said advance. Root causes stacked: the contract read "unrepaired[] for the audit to weigh" AND "the audit does not read the 2.3 report" (self-contradiction); the transport was an inline emit line the host's label-keyed grep silently dropped; and the audit schema had no field proving the evidence was ever weighed. The repair separates FACTS from NARRATIVE: blindness still covers the 2.3 trace/verdict/patches (anti-anchoring), but blocking entries are executed evidence and travel as a self-contained file (`blocking_findings.json`: verbatim step quote + script excerpt + measured numbers per entry) that `next` lists as an audit input. The audit must disposition each entry; refutation requires a concrete flaw (formalization-vs-text mismatch / arithmetic error / illegal instance) — re-reasoning does not refute, because the entire point of the gate is that fluent reasoning misses what only execution surfaces. The `next` guard is deliberately presence/coverage-only (bounce when dispositions are missing or advance coexists with an upheld finding): a deterministic layer cannot judge refutation quality, and pretending it can would dilute the prompt-level standard that actually carries it. Known residual: a lazy audit could refute-all with weak bases; the defense is the refutation standard plus the disposition being visible in the report for the host/user to inspect.

## Why the dual-reading protocol exists

Formalizing prose into code is itself interpretation. A live case: a step's "removal leaves v unchanged" read literally (|Δv|≈0) made the interference gate provably inert; read charitably (Δv≥0) the gate functioned — two of three findings depended on the reading, one was robust. Silently picking one reading either strawmans the candidate or launders its bug. The protocol makes the interpretation auditable: quote the step verbatim, formalize EVERY defensible reading of an ambiguous operative term, run the trace under each, tag findings reading_robust vs reading_dependent. An ambiguous operative term is always itself a wording finding — whichever reading the author meant, the text must be edited to say it, so ambiguity is never free.

## Why the bottleneck-level retry is conditional (and budgeted at 1 + 1 candidate attempt)

The mechanism retry re-rolls generation under the SAME Phase-1 diagnosis; nothing re-examined the diagnosis itself, so a bottleneck whose surrounding space is occupied burned both attempts and terminated. The repair adds one bottleneck re-diagnosis, but gated on DETERMINISTIC failure attribution: only when BOTH attempts died with `paper_pointed_threat.addressable_via = "unaddressable"` (two independently-generated mechanisms subsumed ⇒ the occupied space indicts the framing, not the generator). Mechanism-level deaths do NOT trigger it — a third mechanism roll has diminishing returns, and the space of sharp bottlenecks per corpus is small (the first diagnosis is usually the sharpest; a forced second is often blander). The re-diagnosis receives the retired bottleneck_statement + both killer papers as negative anchors, keeps the `do_not_generate` exit (an honest "this corpus supports no second sharp bottleneck" beats a manufactured one), and the new bottleneck gets ONE candidate attempt (`.retry_used` stays set) — whole-run worst case 3 gauntlet cycles. The attribution rule is deliberately narrow; widen it only on evidence from real runs, not speculation.

## Why cross-run dedup is soft (and lives in the navigator, not the prompts)

Observed live: two runs on adjacent directions (self-evolving agents; skill self-discovery) independently generated the same mechanism family (held-out coalition Shapley) — the second run burned a full gauntlet cycle re-discovering and re-killing it. The skill is deliberately stateless per-run, so the fix is a read-only SCAN at emit time: `next` lists sibling runs' canonical candidates (title + signature terms) as one soft-anchor input line. Soft, never veto: adjacent directions may legitimately need related mechanisms — then the delta must be explicit, which is exactly the differentiation duty 2.2 already carries. The scan lives in the navigator (not the prompts) because only the host side knows where sibling runs are; the prompt's OPTIONAL block just defines how to weigh the line when present. Sibling candidates are context, never corpus — citing them as literature would fabricate provenance.

## Why refuted dispositions get a bounded re-check (and upheld ones don't)

The disposition guard is presence-only by design (a deterministic layer cannot judge quality). That leaves exactly one gameable path: an audit can dismiss executed evidence by marking findings `refuted` with a hollow basis, and presence-checking cannot tell. The asymmetry: an `upheld` disposition CONCEDES the evidence (no incentive problem); a `refuted` one DISMISSES it — so only refutations get a second opinion. The re-check is a single bounded fresh call that judges the refutation, not the candidate: does the basis name a concrete flaw (formalization-vs-text mismatch / arithmetic error / illegal instance), verified against the finding's own quoted text and numbers? Invalid refutation ⇒ the finding counts as upheld and the audit bounces. Cost profile: zero calls on the common path (no refutations), one call when refutation happens. Stale-recheck loops are prevented by matching recheck verdicts to the CURRENT report's refuted finding_refs.

## Why SKILL.md forbids filtering the `next` emit

The navigator's emit is the single wire carrying step inputs to the host. In the incident above, the load-bearing input (blocking findings) rode an unlabeled continuation line; the host captured the emit with a label-keyed grep and the line vanished — no error anywhere, because nothing downstream verified receipt. Rule: consume the emit whole (`head -c` the block if bounding is needed; never filter by line labels). The file-based handoff plus the disposition guard make this particular loss detectable, but the host rule remains the first line of defense for every OTHER input the emit carries.
