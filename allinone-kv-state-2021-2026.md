## Summary of all searched results

### 1. Overview

使用 6 个查询的并集（KV cache 跨请求复用/转移、latent state 通信、semantic caching、stateful tool-using agents 与外部状态变更、computation-state transfer 与部分重算、多消费者 KV 共享 fan-out），时间窗 2021–2026，在 arXiv 与 Crossref 两个可用源上共得 **104 篇去重论文**（另有 16 条跨源重复被合并）。语料明显呈双峰：一峰是 LLM serving 系统侧的 KV cache 复用/共享/压缩（占约一半），另一峰是应用侧的 semantic caching；真正落在"stateful tool-using agent + 外部状态变更后复用"的论文很少。

### 2. Trends

- **时间上明显后移**：2021–2022 的命中多为噪声（quantum state transfer、电力变压器等字面匹配）；真正的主题论文几乎全部出现在 2023 之后，2025–2026 爆发（约 70% 的结果在 2025+）。
- **方法演进**：2023–2024 以前缀精确复用（Prompt Cache [20]、RadixAttention）为主；2024–2025 转向跨请求/跨租户/跨模型共享（KVShare [4]、DroidSpeak [5]、Oneiros [76]）；2026 出现多 agent 集体共享（TokenDance [30]、PolyKV [32]、SparseX [24]）与语义缓存的校准/验证问题（[11]、[14]）。
- **发表形态**：arXiv 占绝对主导；正式会议收录集中在系统会议（NDSS、SoCC、INFOCOM、IPDPS、FPGA）而非 ML 主会，说明该主题目前主要是 systems community 在推动。
- **安全问题刚刚起步**：KV 共享的副作用（prompt 泄漏 [63]、bit-flip [99]）2025 后才出现，属于"复用的安全边界"而非"复用的有效性条件"。

### 3. Key themes（按用户要求的 7 类区分）

1. **Exact cache（精确前缀/KV 复用）** — 相同前缀的逐位复用，有效性由"前缀相同"自动保证，不涉及 receiver-conditioned 判断。代表：[20] Prompt Cache、[25] TraCT、[38] Mell、[87] PRESERVE、[48] KV transfer；Model Knowledge 的 RadixAttention [M2]、ChunkAttention [M6]、vLLM [M1]。
2. **Semantic cache（响应级语义复用）** — 用 embedding 相似度复用整个响应；有效性判断是"问题语义等价"，与计算状态无关。代表：[2]、[11]、[14]、[26]、[88] MeanCache、[89]、[91]、[96] GuardCache；[M7] GPTCache。
3. **Latent/KV state communication（跨模型隐状态通信）** — 把 KV/激活作为消息在模型间传递，不经文本。代表：[5] DroidSpeak、[70] Knowledge Packs；[M10] Cache-to-Cache (uncertain)。
4. **Computation-state transfer（计算状态搬运/重组）** — KV cache 的压缩、传输、拼接与跨请求移植。代表：[1]、[3] KVPR、[8] KVLink、[9] C²KV、[19] KV-CAR、[7] LLMCache；[M3] CacheGen、[M4] CacheBlend。
5. **Receiver-specific validity（接收方条件化的有效性）** — **语料中没有任何一篇直接回答**：现有工作要么假设"同前缀即有效"（exact），要么假设"语义相似即有效"（semantic cache），要么只处理搬运正确性（transfer）。最接近的仅是安全视角：[63]（共享 KV 导致跨租户泄漏——有效性/隔离性的反面）、[99]（共享 KV 块 bit-flip 脆弱性）、[45] ChainCaps（工具链组合后的能力衰减，防越权而非复用有效性）。这正是明确的空白点：普通 prefix cache / response cache / tool cache **均未**回答 receiver-conditioned validity。
6. **Partial repair（部分修复/选择性重算）** — 复用大部分状态、只重算失效部分。代表：[3] KVPR（I/O 感知的部分重算）；[M4] CacheBlend（对复用 chunk 选择少量 token 重算以恢复精度）。数量极少，且都局限在"同一模型、RAG 场景"，没有推广到 receiver 变化或外部状态变更的情形。
7. **真实 multi-consumer / fan-out（同一状态被多个消费方使用）** — 共享前缀被多请求/多租户/多 agent 同时消费。代表：[30] TokenDance、[32] PolyKV、[24] SparseX、[4] KVShare、[5] DroidSpeak、[29] Cascade；[M1] vLLM parallel sampling、[M5] Hydragen、[M6] ChunkAttention。注意这些 fan-out 全部建立在"共享前缀之后各 consumer 独立续写"的假设上，没有处理 consumer 之间外部状态互相污染（一个 consumer 的工具调用改变共享状态后其余 consumer 的复用有效性）的问题。

另外一组与主题正交但相关：**stateful tool-using agents** — [46] ClawVM（harness 管理的 agent 虚拟内存）、[21]（MCP 信息保真度的鞅分析）、[13]（agent 计划复用）、[75]、[86]；[M8] ToolSandbox、[M9] AppWorld（stateful 基准，显式建模外部状态变更）。

### 7. Recommendations for reading（基础 → 前沿）

1. **Prompt Cache [20]（2023）+ RadixAttention/SGLang [M2]（2024）** — exact cache 的两个基线，先弄清"前缀相同 ⇒ 复用有效"这一隐含假设的确切边界。
2. **CacheBlend [M4]（2024）** — partial repair 的唯一成熟样本：复用 + 选择性重算恢复有效性，是向 receiver-conditioned validity 推广的直接出发点。
3. **DroidSpeak [5]（2024）** — 跨模型 KV 共享，说明"同一状态对不同 receiver 的可移植性"已被工程化，但有效性条件未形式化。
4. **KVShare [4]（2025）与 TokenDance [30]（2026）** — 多租户/多 agent 的真实 fan-out 复用，展示 multi-consumer 场景的当前上限。
5. **I Know What You Asked [63]（2025, NDSS）** — 共享 KV 的失效/泄漏面，是定义"复用何时不安全/无效"的最佳反例文献；与 ClawVM [46]、ToolSandbox [M8] 一起构成"stateful agent + 外部状态变更"方向的入口。

**核心结论**：在你的 7 分类中，1/2/3/4/7 已有较成熟文献；6（partial repair）仅 CacheBlend/KVPR 两篇雏形；**5（receiver-conditioned validity）在 2021–2026 的 API 语料中没有直接答案**——所有现有 cache 的有效性判据都是 producer-side 或 content-side 的（前缀相同、语义相似），没有任何工作把"复用有效性"建模为接收方（receiver 的模型、上下文、外部状态视图）的函数，更没有处理 external-state mutation 后多 consumer 的安全复用。这是一个可辩护的空白点。
